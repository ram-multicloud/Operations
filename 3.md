# AWS Systems Manager 
## Topic: Enabling SSM | IAM Role | Fleet Manager | Run Command

## example
Your company has **50 EC2 instances** spread across the organisation.
A **security audit** says every machine must have `nginx` installed and a `devops` user created — by tomorrow morning.

**Bad approach:** SSH into each machine one-by-one → takes hours, error-prone.

**Smart approach:** Use **AWS Systems Manager Run Command** → send one script,
it runs on all 50 machines simultaneously in seconds.

**But first** — AWS needs *permission* to talk to those EC2 instances.
That permission comes from an **IAM Role**.

---

## IAM Role for Systems Manager

### IAM Role:

An **IAM Role** is a set of permissions that AWS services can *assume*.
Think of it like a **security badge** — it tells AWS Systems Manager:

> *"You are allowed to talk to this EC2 instance and run commands on it."*

### EC2 Need a Role for SSM

```
┌─────────────────────────────────────────────────────────────┐
│                   Without IAM Role                          │
│                                                             │
│  AWS Systems Manager ──X──► EC2 Instance                   │
│                    (Access Denied!)                         │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    With IAM Role                            │
│                                                             │
│  AWS Systems Manager ──✅──► EC2 Instance                  │
│         (Role: AmazonSSMManagedInstanceCore attached)       │
└─────────────────────────────────────────────────────────────┘
```

---

### Create an IAM Role for EC2 + SSM

**Select Trusted Entity:**

| Field | Value |
|-------|-------|
| Trusted entity type | **AWS Service** |
| Use case | **EC2** |

** Attach Permissions Policy:**

Search for and select:
```
AmazonSSMManagedInstanceCore
```

> This is an **AWS Managed Policy** — pre-built by AWS.  
> It grants the SSM Agent on the EC2 permission to communicate with Systems Manager.

**permissions:

```json
{
  "Effect": "Allow",
  "Action": [
    "ssm:DescribeAssociation",
    "ssm:GetDocument",
    "ssm:GetParameters",
    "ssm:ListAssociations",
    "ssm:PutInventory",
    "ssm:UpdateAssociationStatus",
    "ssm:UpdateInstanceInformation",
    "ssmmessages:CreateControlChannel",
    "ssmmessages:CreateDataChannel",
    "ssmmessages:OpenControlChannel",
    "ssmmessages:OpenDataChannel",
    "ec2messages:AcknowledgeMessage",
    "ec2messages:SendReply"
  ]
}
```


**Name the Role:**

| Field | Suggested Value |
|-------|----------------|
| Role name | `EC2-SSM-Role` |
| Description | `Allows EC2 instances to communicate with AWS Systems Manager` |

Click **Create Role** ✅


### IAM Role vs IAM User vs IAM Policy

| Concept | What It Is | Real-World Analogy |
|---------|-----------|-------------------|
| **IAM User** | A person (developer, admin) | An employee with a badge |
| **IAM Role** | A set of permissions worn by a *service* | A security clearance for a machine |
| **IAM Policy** | The actual list of allowed/denied actions | A rulebook |
| **Instance Profile** | The container that holds a role for EC2 | The ID card holder |


## Attach the Role to an EC2 Instance

### Option A — Attach Role at EC2 Launch Time

During EC2 creation:
```
EC2 Console → Launch Instance
→ Advanced Details
→ IAM Instance Profile → Select: EC2-SSM-Role
```

### Option B — Attach Role to an Existing EC2 Instance

```
EC2 Console
→ Select the Instance
→ Actions → Security → Modify IAM Role
→ Select: EC2-SSM-Role
→ Update IAM Role ✅
```

## Fleet Manager — Viewing All Managed Nodes

Once the role is attached and the SSM Agent is running, the EC2 instance
appears in **Fleet Manager** — AWS's dashboard for all managed nodes.

### Navigate to Fleet Manager:
```
AWS Console → Systems Manager → Node Management → Fleet Manager
```

### Architecture — How SSM Agent Communicates:

```
┌──────────────────────────────────────────────────────────────┐
│                        AWS Cloud                             │
│                                                              │
│   ┌──────────────────┐        ┌───────────────────────────┐  │
│   │  Systems Manager │◄──────►│  EC2 Instance             │  │
│   │  Service (Cloud) │        │  ┌─────────────────────┐  │  │
│   └──────────────────┘        │  │  SSM Agent (waits   │  │  │
│                               │  │  for instructions)  │  │  │
│   ┌──────────────────┐        │  └─────────────────────┘  │  │
│   │  Fleet Manager   │        │                           │  │
│   │  (Dashboard)     │        │  IAM Role: EC2-SSM-Role   │  │
│   └──────────────────┘        └───────────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

> The SSM Agent communicates **outbound over HTTPS (port 443)**.
> You do NOT need to open port 22 (SSH) at all!


### checking SSM Agent Status on EC2

If an instance is NOT showing in Fleet Manager, check agent status:

**Amazon Linux / RHEL:**
```bash
sudo systemctl status amazon-ssm-agent
sudo systemctl start amazon-ssm-agent       # if not running
sudo systemctl enable amazon-ssm-agent      # start on boot
```

**Ubuntu:**
```bash
sudo systemctl status snap.amazon-ssm-agent.amazon-ssm-agent
```

---

### Installing SSM Agent on Custom/Older AMIs

Amazon Linux 2 and Amazon Linux 2023 have SSM Agent **pre-installed**.
For custom or older images, install manually:

**Amazon Linux / RHEL / CentOS:**
```bash
sudo dnf install -y amazon-ssm-agent
sudo systemctl enable amazon-ssm-agent
sudo systemctl start amazon-ssm-agent
```

**Ubuntu:**
```bash
sudo snap install amazon-ssm-agent --classic
sudo systemctl enable snap.amazon-ssm-agent.amazon-ssm-agent
sudo systemctl start snap.amazon-ssm-agent.amazon-ssm-agent
```

> Reference: [AWS Docs – Manually Install SSM Agent on Linux](https://docs.aws.amazon.com/systems-manager/latest/userguide/manually-install-ssm-agent-linux.html)



## Run Command — Run Scripts on Multiple EC2s at Once

**Run Command** is the SSM feature that lets you run a shell script on
**one or thousands of EC2 instances simultaneously** — no SSH needed.

### Navigate to Run Command:
```
AWS Console → Systems Manager → Node Management → Run Command → Run Command
```

### Run Command Workflow:

```
Admin selects Command Document
         │
         ▼
Choose target instances (by tag, ID, or all in Fleet)
         │
         ▼
Write/paste the shell script
         │
         ▼
Click "Run" → SSM sends to all selected EC2s in parallel
         │
         ▼
View output per instance in real-time
```

### Most Used Command Document:

| Document Name | Use |
|--------------|-----|
| `AWS-RunShellScript` | Run any bash script on Linux |
| `AWS-RunPowerShellScript` | Run PowerShell on Windows |
| `AWS-UpdateSSMAgent` | Update SSM Agent on instances |
| `AWS-InstallWindowsUpdates` | Windows patching |

### ✅ Exercise 2: Install nginx on Both Machines

**Goal:** Install `nginx` on 2 EC2 instances using a single Run Command.

**Navigate:**
```
Systems Manager → Run Command → Run Command
→ Document: AWS-RunShellScript
→ Targets: Select both EC2 instances
```

**Script (for Amazon Linux):**
```bash
#!/bin/bash
sudo dnf update -y
sudo dnf install -y nginx
sudo systemctl start nginx
sudo systemctl enable nginx
echo "nginx installed on $(hostname)"
```

> The blog post script uses `apt install nginx` — that is for **Ubuntu**.
> Since the exercise uses **Amazon Linux**, use `dnf install nginx` instead.

| OS | Package Manager | Install Command |
|----|----------------|----------------|
| **Ubuntu** | apt | `apt install -y nginx` |
| **Amazon Linux 2023** | dnf | `dnf install -y nginx` |
| **RHEL / CentOS 8+** | dnf | `dnf install -y nginx` |

**Verify nginx is running:**
```bash
sudo systemctl status nginx
curl http://localhost
```


### Exercise 3: Create a User Called `devops` on Both Machines

**Goal:** Create the `devops` user on all EC2 instances at once.

**Navigate:**
```
Systems Manager → Run Command → AWS-RunShellScript
→ Targets: Select both instances
```

**Script:**
```bash
#!/bin/bash

# Create the devops user
sudo useradd -m -s /bin/bash devops

# Verify user was created
id devops

# Add devops to sudoers (optional — admin privilege)
echo "devops ALL=(ALL) NOPASSWD:ALL" | sudo tee /etc/sudoers.d/devops

echo "User 'devops' created successfully on $(hostname)"
```

**Expected Output (per instance):**
```
uid=1001(devops) gid=1001(devops) groups=1001(devops)
User 'devops' created successfully on ip-172-31-xx-xx
```

> 🎙️ *Explain `useradd` flags:*

| Flag | Meaning |
|------|---------|
| `-m` | Create home directory `/home/devops` |
| `-s /bin/bash` | Set bash as the default shell |

### ✅ Exercise 4: Uninstall git from Both Machines

**Goal:** Remove `git` from all EC2 instances in one command.

**Script (Amazon Linux):**
```bash
#!/bin/bash
sudo dnf remove -y git
echo "git uninstalled from $(hostname)"

# Verify removal
if ! command -v git &> /dev/null; then
    echo "Confirmed: git is NOT installed"
else
    echo "Warning: git still found"
fi
```

**Script (Ubuntu — if using Ubuntu EC2s):**
```bash
#!/bin/bash
sudo apt remove -y git
sudo apt autoremove -y
echo "git uninstalled from $(hostname)"
```

### 🌟 Stretch Exercise: Full Setup Script (Combine All Steps)

**Script — Install nginx + Create devops user + Remove git:**
```bash
#!/bin/bash
set -e   # Exit immediately if a command fails

echo "===== Starting Setup on $(hostname) ====="

# 1. Update packages
sudo dnf update -y
echo "[✅] Packages updated"

# 2. Install nginx
sudo dnf install -y nginx
sudo systemctl start nginx
sudo systemctl enable nginx
echo "[✅] nginx installed and started"

# 3. Create devops user
if id "devops" &>/dev/null; then
    echo "[ℹ️] User 'devops' already exists — skipping"
else
    sudo useradd -m -s /bin/bash devops
    echo "[✅] User 'devops' created"
fi

# 4. Uninstall git
sudo dnf remove -y git
echo "[✅] git uninstalled"

echo "===== Setup Complete on $(hostname) ====="
```

> 🎙️ *Point out `set -e` — if any step fails, the script stops immediately.
> This prevents half-configured machines in production.*

## 🧠 Concept Check — Quiz Questions

**Q1:** What policy must be in the IAM Role for SSM to work?
> *Answer: `AmazonSSMManagedInstanceCore`*

> *Answer: No — SSM communicates over HTTPS (port 443) outbound only.*

**Q3:** Where in the AWS Console do you see all SSM-managed EC2 instances?
> *Answer: Systems Manager → Fleet Manager*

**Q4:** What command document do you use to run a bash script via Run Command?
> *Answer: `AWS-RunShellScript`*

**Q5:** What is the correct install command for nginx on Amazon Linux 2023?
> *Answer: `sudo dnf install -y nginx`*

**Q6:** Why can't you use `apt install` on Amazon Linux?
> *Answer: Amazon Linux uses `dnf` (RPM-based). `apt` is for Debian/Ubuntu.*

## 🔧 Troubleshooting — Instance Not Showing in Fleet Manager

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| Instance not in Fleet Manager | IAM Role not attached | Attach `EC2-SSM-Role` via Actions → Security |
| Instance not in Fleet Manager | SSM Agent not running | Run `systemctl start amazon-ssm-agent` |
| Instance not in Fleet Manager | No internet / VPC endpoint | Check Security Group allows outbound 443 |
| Instance not in Fleet Manager | Wrong region | Make sure you're in the same region as EC2 |
| Run Command times out | Agent not responding | Restart SSM agent, check IAM permissions |
